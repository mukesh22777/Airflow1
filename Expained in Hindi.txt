@task.short_circuit(task_id="check_file_availability")

    # 1. Config load karo (dev.yaml/uat.yaml/prod.yaml)
    # 2. Train.csv file exist karti hai?

Hindi mein: Sabse pehle check hota hai ki Train.csv file available hai ya nahi. Agar file nahi hai ya 24 ghante se update nahi hui, toh pipeline ruk jaati hai aur message print hota hai: "File is not available to process".

Phase 2: Bronze Ingest (Raw Data Load)
File: jobs/bronze_ingest.py

Kya karta hai:

✅ Config se source path padho (C:/data/Train.csv)
✅ Watermark check karo — "Pichli baar kab tak ka data load hua tha?"
✅ Full Load (pehli baar): Poori file read karo
✅ Incremental Load (baad mein): Sirf naye rows (jinka Load_date > last watermark)
✅ Schema validate karo (35 columns sahi hain?)
✅ Load_date column add karo (current timestamp)
✅ Data ko JSON payload mein convert karo
✅ Train_bronze Hive table mein write karo (Parquet format)
Bronze table mein kya store hota hai:

| payload (JSON)          | _ingest_ts | _run_id      | _source_table     | ingest_date |
|-------------------------|------------|--------------|-------------------|-------------|
| {"Age":41,"Attrition":..}| 2026-05-05 | 20260505220000| filesystem.Train.csv| 2026-05-05 |
Kyun JSON payload?
Agar future mein source file mein columns change hon toh Bronze break nahi hoga — raw data safe rehta hai.

Phase 3: Silver Transform (Cleaning)
File: jobs/silver_transform.py

Kya karta hai:

✅ Bronze table se current run ka data read karo
✅ JSON payload ko wapas columns mein parse karo
✅ Drop Duplicates — same rows hataao
✅ Fill NA with 0 — null values ko 0 se replace karo
✅ Select relevant columns — sirf zaroori 35 columns rakho
✅ Load_date retain karo (watermark ke liye)
✅ Train_silver Hive table mein write karo (Parquet)

⚡ Step 3: Full Load vs Incremental Load — Fark Samjho
Full Load (Pehli Baar)
Day 1: Koi watermark nahi hai
    ↓
Poori Train.csv padho (saare rows)
    ↓
Bronze mein store karo
    ↓
Watermark save karo: "Load_date = 2026-05-05 22:00:00"
Incremental Load (Baad ke Runs)
Day 2: Watermark = "2026-05-05 22:00:00"
    ↓
Sirf woh rows padho jinka Load_date > 2026-05-05 22:00:00
    ↓
Bronze mein APPEND karo (puraane data ke saath)
    ↓
Watermark update: "Load_date = 2026-05-06 22:00:00"
Fayda: Roz poora data process nahi karna padta — sirf naya data process hota hai = faster + cheaper.

🧠 Step 4: Watermark Kaise Kaam Karta Hai
Control Table (control.etl_watermark_state):

| environment | source_table | watermark_column | last_success_value   | last_status |
|-------------|-------------|-----------------|---------------------|-------------|
| dev         | Train       | Load_date       | 2026-05-05 22:00:00 | SUCCESS     |
| uat         | Train       | Load_date       | 2026-05-05 22:00:00 | SUCCESS     |
| prod        | Train       | Load_date       | 2026-05-05 22:00:00 | SUCCESS     |
Key rule: Watermark sirf Gold success ke baad update hota hai:

Agar Bronze/Silver fail ho → watermark NAHI badlega
Next run same data phir se process karega
Data loss kabhi nahi hoga!
Environments isolated hain: DEV ka watermark PROD ko affect nahi karega.

🗓️ Step 5: Airflow DAG — Orchestration Samjho
File: dags/sqlserver_hive_medallion_dag.py

@dag(
    dag_id="sqlserver_hive_medallion_dag",
    schedule="0 22 * * *",   # ← Roz raat 10 PM
    catchup=False,            # ← Purane din ka data process mat karo
    max_active_runs=1,        # ← Ek time pe ek hi run
)
def sqlserver_hive_medallion_dag():

    # Task 1: File available hai?
    gate = check_file_availability()

    # Task 2: Bronze (raw load)
    bronze_run_id = run_bronze_ingest()

    # Task 3: Silver (clean)
    silver_run_id = run_silver_transform(bronze_run_id)

    # Task 4: Gold (aggregate)
    run_gold_aggregate(silver_run_id)

    # Order: gate → bronze → silver → gold
    gate >> bronze_run_id
TaskFlow API (@dag, @task) ka fayda:

Code cleaner hai (purane BashOperator se)
Tasks ke beech data pass karna easy hai (XCom automatic)
Airflow 2.x ka latest recommended pattern

🛡️ Step 8: Error Handling & Validation
Har step pe validation hoti hai:

Check	Kahan	Kya Karta Hai
File exists?	DAG (first task)	File nahi → pipeline ruki
Schema valid?	Bronze	35 columns match nahi → ERROR
Empty data?	Silver/Gold	Koi row nahi → skip (no error)
Duplicates?	Silver	dropDuplicates() se remove
Null values?	Silver	fillna(0) se fix
Watermark safe?	Gold	Sirf success pe update
Agar koi step fail ho:

Airflow 1 retry karega (5 min baad)
Phir bhi fail → alert/email
Watermark safe hai — next run same data phir process karega
